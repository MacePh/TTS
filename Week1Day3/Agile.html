A Brief History / The Waterfall Method

Before Agile, Waterfall was the industry standard method for developing and delivering software.

Waterfall is:

  • A strict linear method of sequential phases (requirements, design, implementation, verification, and maintenance)
  • Each phase must be 100% complete before the next phase begins

Introduction To Agile

Waterfall: Pros

  • Easy to understand
  • Works well for projects with clear objectives and stable requirements
  • Good for less experienced teams or project managers
  • Good for teams whose composition changes frequently

Introduction To Agile

Waterfall: Cons

  • Projects could not be deployed until they were 100% complete
  • Led to increasing Application Delivery Lag

 

Application Delivery Lag: The time between business needs being validated and production-ready software.

 

Introduction To Agile

Enter... Agile

Agile is based on collaborative decision making between requirements and solutions teams that emphasizes iterative progressions

  • allows for changes in the development plan
  • allows for changes in the development pattern
  • short, repetitive, development-feedback cycles 
  • frequent, iterative, deployment

Agile: Pros

  • minimizes risk through iterative development, testing, and evaluation
  • rapid software delivery
  • integrates user feedback into the development cycle

Introduction To Agile

Agile: Cons

  • relies on real-time communication
  • lag in creating new user documentation
  • feedback and iterative release can be time and labor intensive for users 
  • Can be inefficient in large organizations

Agile in Practice: User Stories

Introduction To Agile

User Stories: Descriptions of software functionality told from the perspective of the user. Used to help identify tasks.

Should include:

1. Type of User: the who

2. Some Feature: the what

3. Some Reason: the why

Agile in Practice: User Stories

Introduction To Agile

Writing User Stories:

1. User story, not a user novel

2. 1 piece of functionality only

3. User perspective

Introduction To Agile

Sample outcome: 

1. As a developer, I want to reach an API endpoint with a search query and get back matching results so I can implement a search bar.
2. As a user, I want to input a query in a search bar, and get taken to a page of results based on product name so I can search for products.

Agile in Practice: Kanban Framework

Introduction To Agile

Kanban facilitates continuous development through the use of a Kanban board that displays each project (user story) on a card and moves cards through columns that represent progressive stages of completion.

Introduction To Agile

Agile in Practice: Kanban Framework

  • work-in-progress limits relate to a teams capacity.

 

  • tickets moved out of one column signals a need to add new tickets to another.

Scrum Versus Kanban

  Scrum Kanban
Cadence Regular fixed length sprints (e.g. 2 weeks) Continuous flow
Release methodology At the end of each sprint if approved by the product owner Continuous delivery or at the teams discretion
Roles Product owner, scrum master, development team No existing roles. Some teams enlist the help of an agile coach.
Key metrics Velocity Cycle time
Change philosophy Teams should strive to not make changes to the sprint forecast during the sprint. Doing so compromises learnings around estimation. Change can happen at any time
